# Ideation Frameworks Reference

Use these frameworks selectively. Pick the lens that fits the idea; do not run every framework mechanically. The goal is to unlock thinking, not to clear a checklist.

## SCAMPER

Seven operations for transforming an existing idea:

- **Substitute:** what component, material, or process could you swap out? Replace the core technology? The target audience? The business model?
- **Combine:** what if you merged this with another product, service, or idea? Which two things that do not usually go together would make something new?
- **Adapt:** what else is like this? Which ideas from other industries, domains, or time periods could you borrow? What parallel exists in nature?
- **Modify (magnify/minimize):** make it 10x bigger? 10x smaller? Exaggerate one feature? Strip it to the absolute minimum?
- **Put to other uses:** who else could use this? What other problems could it solve? What happens in a completely different context?
- **Eliminate:** remove a feature entirely? The zero-configuration version? Half the steps?
- **Reverse/rearrange:** run the steps in the opposite order? Have the user do the work instead of the system, or the reverse? Reverse the value chain?

**Best for:** improving or reimagining existing products and features. Weak for greenfield ideas.

## How Might We (HMW)

Reframe a problem as an opportunity in the "How might we..." format:

- Start from an observation or pain point.
- Reframe it as "How might we [desired outcome] for [specific user] without [key constraint]?"
- Generate multiple HMW framings of the same problem — different framings unlock different solutions.

**Good HMW:**
- Narrow enough to be actionable ("...help new users find relevant content in their first 5 minutes").
- Broad enough to allow creative solutions (not "...add a recommendation sidebar").
- Carries a tension or constraint that forces creativity.

**Bad HMW:**
- Too broad: "How might we make users happy?"
- Too narrow: "How might we add a button to the settings page?"
- Solution-embedded: "How might we build a chatbot for support?"

**Best for:** reframing stuck thinking. When someone is anchored on a solution, pull them back to the problem.

## First Principles Thinking

Break the idea down to its fundamental truths, then rebuild:

1. **What is true?** Not assumed, not conventional — actually true.
2. **What are we assuming?** List every assumption, including the ones that feel obvious.
3. **Which assumptions can we challenge?** For each: is this a law of physics, or just how it has been done?
4. **Rebuild from the truths.** Given only the fundamentals, what would you build?

**Best for:** breaking out of incremental thinking. When every idea feels like a small improvement on the status quo.

## Jobs to Be Done (JTBD)

Focus on what the user is trying to accomplish, not what they say they want:

- **Functional job:** what task are they trying to complete?
- **Emotional job:** how do they want to feel?
- **Social job:** how do they want to be perceived?

Format: "When I [situation], I want to [motivation], so I can [expected outcome]."

People do not buy products; they hire them to do a job. The competing product is not always in the same category — Netflix competes with sleep, not just other streaming services.

**Best for:** understanding the real problem. When you are not sure you are solving the right thing.

## Constraint-Based Ideation

Impose constraints deliberately to force creative solutions:

- **Time:** only 1 day to build this?
- **Feature:** only one feature?
- **Tech:** cannot use the obvious technology?
- **Cost:** free forever?
- **Audience:** the user has never used a computer before?
- **Scale:** must work for 1 billion users? For just 10?

**Best for:** cutting through complexity. When the idea is growing too large or too vague.

## Pre-mortem

Assume the idea already failed, then work backwards:

1. It is 12 months from now. The project shipped and flopped. What went wrong?
2. List every plausible failure mode — technical, market, team, timing.
3. For each: is it preventable? Is it a signal the idea needs to change?
4. Which failure modes will you accept? Which ones would kill the project?

**Best for:** Phase 2 evaluation. Stress-testing ideas that feel good but have not been pressure-tested.

## Analogous Inspiration

Look at how other domains solved a similar problem:

- Which industry has already solved a version of this?
- What would this look like if a specific company or product built it?
- What natural system works this way?
- What historical precedent exists?

Find *structural* similarities, not surface ones. "Uber for X" is surface-level. "A two-sided marketplace that solves a trust problem between strangers" is structural.

**Best for:** Phase 1 expansion. Generating variations that differ genuinely from the obvious approach.
